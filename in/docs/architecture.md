# Project Architecture — Campus Facility Booking System

> **Course**: CPEN 412 — Web Software Architecture, University of Ghana  
> **Backend**: Spring Boot 4.0.2 | Java 17 | Maven | Aiven PostgreSQL  
> **Frontend**: Next.js 16.1.6 | React 19 | TypeScript 5  
> **Architecture**: Model–View–Controller (MVC)

---

## Overview

A **Campus Facility Booking System** allowing users to view campus facilities, check availability in 30-minute time slots, and create/manage bookings. Administrators can manage buildings, facilities, users, and monitor bookings via a dashboard with analytics.

## How MVC Is Implemented

### Model Layer
- **JPA Entities** (`com.book.in.model`): `Building`, `Facility`, `User`, `Booking`, `BookingStatus`, `FacilityType`, `AvailabilitySlot`
- Entities map directly to PostgreSQL tables via Hibernate annotations (`@Entity`, `@Table`, `@ManyToOne`, `@JoinColumn`)
- Validation annotations (`@NotBlank`, `@NotNull`, `@Email`, `@Positive`) enforce data integrity at the model level
- The database schema is auto-generated by Hibernate's `ddl-auto=update`

### View Layer
- **Next.js frontend** serves as the View — it renders the user interface
- React components consume REST API responses and present data to users
- No server-side rendering of HTML from Spring Boot — the backend is a pure REST API
- The View never directly accesses the database; all data flows through Controllers

### Controller Layer
- **REST Controllers** (`com.book.in.controller`): `FacilityController`, `BookingController`, `AvailabilityController`, `AuthController`, `UserController`, `BuildingController`
- Controllers receive HTTP requests, delegate to Service classes for business logic, and return JSON responses
- Controllers handle validation, error responses, and appropriate HTTP status codes
- `@RestControllerAdvice` (`GlobalExceptionHandler`) provides centralized error handling

### Service Layer (Business Logic)
- **Services** (`com.book.in.service`): `BookingService`, `FacilityService`, `UserService`
- Services contain business logic: booking conflict detection, availability slot generation, password hashing
- Services use Repository interfaces for data access — Controllers never access Repositories directly

### Repository Layer (Data Access)
- **Spring Data JPA Repositories** (`com.book.in.repository`): `BookingRepository`, `FacilityRepository`, `UserRepository`, `BuildingRepository`
- Extend `JpaRepository` for standard CRUD operations
- Custom `@Query` methods for specialized queries (e.g., `findConflictingBookings`)

## Database Configuration

Secrets are stored in `.env` (gitignored). Spring Boot loads them via `spring.config.import`.

| Variable | Description |
|----------|-------------|
| `DB_HOST` | Aiven PostgreSQL host |
| `DB_PORT` | PostgreSQL port |
| `DB_NAME` | Database name |
| `DB_USERNAME` | Database username |
| `DB_PASSWORD` | Database password |
| `CORS_ORIGIN` | Allowed frontend origin (default: `http://localhost:3000`) |

---

## Database Schema (3NF)

All tables follow **3rd Normal Form** — no transitive dependencies, all non-key attributes depend solely on the primary key.

| Entity | Table | Key Fields | Relationships |
|--------|-------|-----------|---------------|
| `Building` | `buildings` | `id`, `name`, `code`, `campus`, `description` | Has many Facilities |
| `Facility` | `facilities` | `id`, `name`, `building_id` (FK), `type`, `location`, `capacity` | Belongs to Building, Has many Bookings |
| `User` | `users` | `id`, `name`, `email` (unique), `role`, `password` | Has many Bookings |
| `Booking` | `bookings` | `id`, `facility_id` (FK), `user_id` (FK), `date`, `start_time`, `end_time`, `status`, `purpose` | Belongs to Facility, Belongs to User |

## Key Features (Beyond Requirements)

- **Building Management**: Hierarchical organization (Buildings → Facilities)
- **Admin Dashboard**: Full CRUD for buildings, facilities, users + analytics
- **Personal Insights**: Usage analytics, favorite rooms, weekly activity
- **Availability Grid**: Visual 30-minute slot picker (08:00–20:00)
- **Booking Conflict Detection**: Server-side conflict checking with JPQL query
- **Password Security**: BCrypt hashing, `@JsonIgnore` on password field
- **Swagger UI**: Interactive API documentation via SpringDoc OpenAPI
- **Loading States**: Skeleton loaders and submit-in-progress indicators
- **Responsive Design**: Mobile-friendly with hamburger navigation
- **Data Seeding**: Auto-populates demo data (10 buildings, 25 facilities, 4 users)
- **LocalStorage Caching**: Frontend caches API responses with TTL
